## Day1：明确模型边界与实际输出链路

### 目标

把你负责的小节范围说清楚，避免后续公式、参数和代码没有边界。

### 需要完成

1. 明确本模型在总系统中的位置：

```text
能源岛可用电力
    ↓
分配到多类价值转化通道：
1. 电力外送：电力 -> 海底电缆 -> 陆上电网
2. 绿色算力：电力 -> 海上/海底数据中心 -> 海底光缆 -> 数据服务输出
3. 氢气输出：电力 -> 电解制氢 -> 储氢 -> 管道/船运 -> 陆上或用氢场景
4. 海洋综合用能：电力 -> 岛上辅助负荷/海水淡化/海洋装备供能
```

2. 写清三条主链路的实际含义：

- 电力外送不是无限送出，受海缆容量、距离、输电损耗、陆上并网消纳能力约束。
- 算力输出不是“电力通过光缆传输”，而是电力在岛上被数据中心消耗，计算结果或数据服务通过光缆输出。
- 氢气输出不是只算制氢量，还要区分管道输氢和船运输氢两种输出方式。

3. 暂定基础尺度：

- 时间尺度：小时级调度，时间步为 `t`。
- 场景长度：先支持 24h 典型日和 168h 典型周。
- 空间尺度：能源岛到陆上接入点距离记为 `L`。
- 电缆、光缆、管道、船运均与距离有关。

4. 明确暂不展开的复杂内容：

- 不做详细电网潮流计算，只做外送容量和损耗约束。
- 不做复杂通信网络路由，只做光缆带宽、延迟和可用性约束。
- 不做船舶逐航次精细调度，Day1-Day3 先用等效运输能力表示。

## Day3：整理符号、约束和基础公式

### 目标

形成可以写进第四章的小节雏形。

### 统一符号

| 符号 | 含义 | 单位 |
|---|---|---|
| `P_grid,t` | t 时刻通过海缆外送的电功率 | MW |
| `P_grid_del,t` | t 时刻陆上实际接收电功率 | MW |
| `P_compute,t` | t 时刻算力中心总用电功率 | MW |
| `P_it,t` | t 时刻 IT 设备用电功率 | MW |
| `B_fiber,t` | t 时刻光缆占用带宽 | Gbps/Tbps |
| `P_h2_el,t` | t 时刻电解制氢用电功率 | MW |
| `H_prod,t` | t 时刻制氢量 | kg/h |
| `H_pipe,t` | t 时刻管道输氢量 | kg/h |
| `H_ship,t` | t 时刻等效船运输氢量 | kg/h |
| `H_storage,t` | t 时刻储氢量 | kg |
| `P_marine,t` | t 时刻海洋综合用能功率 | MW |

### 电力-海缆基础约束

```text
0 <= P_grid,t <= min(P_grid_max, P_grid_accept_max)
```

```text
P_grid_del,t = P_grid,t * (1 - loss_cable)
```

```text
R_power,t = P_grid_del,t * price_power
```

### 算力-光缆基础约束

```text
P_compute_min <= P_compute,t <= P_compute_max
```

```text
P_compute,t = P_it,t * PUE
```

```text
0 <= B_fiber,t <= B_fiber_max
```

```text
R_compute,t = P_it,t * price_compute
```

如果后续拿不到算力服务价格，可先用“单位 IT 电耗收益”近似；如果拿到 GPU 小时价格，再换成 GPU 小时模型。

### 氢气-管道/船运基础约束

```text
H_prod,t = P_h2_el,t * eta_el
```

```text
H_prod,t = H_pipe,t + H_ship,t + ΔH_storage,t
```

```text
0 <= H_pipe,t <= H_pipe_max
```

```text
0 <= H_ship,t <= H_ship_max
```

```text
0 <= H_storage,t <= H_storage_max
```

```text
R_h2,t = (H_pipe,t + H_ship,t) * price_h2
```

### 接入总功率平衡

你负责的模型需要接入总功率平衡：

```text
P_available,t =
P_grid,t
+ P_compute,t
+ P_h2_el,t
+ P_marine,t
+ P_storage_ch,t
+ P_curt,t
```

其中你主要负责：

```text
P_grid,t + P_compute,t + P_h2_el,t + P_marine,t
```

## 总结

Day1-Day3 的核心是把“模型该算什么”讲清楚，包括边界、参数、符号和基础公式。

Day4-Day7 的核心是把它变成“能被代码表达、能被队友接入、能被技术总监审核”的最小可运行模型。

你不要一开始就追求完整复杂优化算法，而应优先保证：

- 工程链路真实
- 参数口径清楚
- 变量符号统一
- 约束条件完整
- 模型能接入总功率平衡
- 代码能跑通最小样例
