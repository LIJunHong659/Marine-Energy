# 蓝海枢纽接口统一协议

本文件是本仓库的项目级协作说明。每次进入本项目工作时，先阅读本文件，再阅读与任务直接相关的源码或文档。

## 1. 总原则

- 4.3--4.7 联合运行必须使用同一个公共参数包 `common_case_v1`。
- 模块内部默认参数只允许用于单元测试和接口冒烟测试，不能用于正式联合运行或工程经济结论。
- 所有联合运行结果必须携带 `caseId=common_case_v1`。
- 缺少 P0 级参数时，禁止运行 `engineering_base`；不能自动回退到模块默认值。
- 接口联调和工程基准分开：
  - `interface_smoke`：只用于 4.3--4.7 接口联调，不形成工程经济结论。
  - `engineering_base`：用于 8760 小时运行、经济性分析和方案比较，必须由场址和企业数据校准。

## 2. 公共参数包结构

推荐目录结构：

```text
4.1边界与口径/
├─ 蓝海枢纽_4.1边界与口径冻结表_V1.1.docx
└─ common_case_v1/
   ├─ common_case_v1.yaml
   ├─ source_timeseries.csv
   ├─ availability.csv
   ├─ demand_and_price.csv
   ├─ initial_state.yaml
   └─ common_case_v1.mat
```

文件职责：

- `common_case_v1.yaml`：静态参数、容量、口径、版本、场景类型。
- `source_timeseries.csv`：4.3 资源与源侧出力时序。
- `availability.csv`：全模块设备可用状态、降额、检修和故障标签。
- `demand_and_price.csv`：负荷、需求、价格和结算口径。
- `initial_state.yaml`：SOC、储氢量、算力队列、设备启停状态等。
- `common_case_v1.mat`：MATLAB 读取版本，必须与 YAML/CSV 数据一致。

## 3. 必须冻结的公共字段

- `caseId`：固定为 `common_case_v1`。
- `scenarioType`：只能是 `interface_smoke` 或 `engineering_base`，单次运行只能选一个。
- `timezone`：固定为 `Asia/Shanghai`。
- `timeInterval`：时段定义为 `[t, t + Delta_t)`。
- `siteId`：同一场址、同一登陆点、同一公共母线拓扑。
- `sourceCapacity`：源侧装机必须由 4.1 冻结。接口联调可暂用 34 MW 风电 + 4 MW 光伏 + 2 MW 潮流能，共 40 MW，并标注为假设值。
- `bessPowerMW`、`bessEnergyMWh`：由 4.5 提供，4.1 冻结。
- `electrolyzerMW`、`h2TankKg`、`SEC`、初始氢库存：由 4.5 提供，4.1 冻结。
- `dcFacilityMW`、IT 容量、基础负荷、任务队列参数：由 4.6 提供，4.1 冻结。
- 海缆送端容量、受端接纳、距离、损耗模型、既有/新建标识：由 4.7 提供，4.1 冻结比较口径。
- 经济日历：币值基准年建议 2026 年，投运年记为 `Y0`，主评价期建议 25 年；这些均为待校准假设。
- 随机种子：联调场景统一固定，保证方案可复现。

## 4. 测点定义

所有接口字段必须显式绑定测点，禁止使用含糊的 `available_power_mw`、`wind_power_mw` 等名称而不说明位置。

| 测点 | 含义 | 责任模块 |
| --- | --- | --- |
| `MP-DEVICE` | 风机、光伏或潮流能设备电气端毛功率 | 单源模型 |
| `MP-03-SOURCE-POI` | 扣除源侧集电损耗后的多源汇集功率 | 4.3 |
| `MP-04-COMMON-BUS` | 储能、制氢、算力、外送等共同连接的功率平衡点 | 4.4 |
| `MP-CABLE-SEND` | 海缆海上送端 | 4.7 |
| `MP-CABLE-RECEIVE` | 陆上受端/结算点 | 4.7 |
| `MP-DC-FACILITY` | 数据中心设施总用电入口 | 4.6 |
| `MP-ELECTROLYZER` | 电解制氢系统用电入口 | 4.5 |
| `MP-H2-DELIVERY` | 管道/船运损耗后的氢产品交付点 | 4.7 |

当前约定：

- 4.3 输出 `pActualAtPOI` 位于 `MP-03-SOURCE-POI`，单位为 W，已扣除集电损耗，尚未扣除源侧辅机。
- 4.4 使用源侧实际注入为正值，同时将 `pSourceAuxLoad` 作为独立负荷扣除一次。
- 4.7 的 `exported_power_mw` 是 `MP-CABLE-SEND`，`delivered_power_mw` 是 `MP-CABLE-RECEIVE`。送端功率进 4.4 平衡，受端电量只用于收益/交付。

## 5. 功率正方向

电池优先使用两个非负变量：

```text
P_BESS_ch >= 0
P_BESS_dis >= 0
```

需要单一净功率接口时再定义：

```text
P_BESS_net = P_BESS_dis - P_BESS_ch
```

其中正值表示向公共母线放电。

其他端口统一使用非负幅值：

- `pSourceActual`：正注入。
- `pGridImport`：正注入。
- `pExportSend`：正消耗。
- `pElectrolyzer`：正消耗。
- `pDCFacility`：正消耗。
- `pMarine`：正消耗。
- `pSpill`：正消耗。
- `pUnserved`：可靠性统计量，不进入实际已服务负荷。

每个可控端口统一保留三层接口：

```text
requested -> accepted -> actual
```

4.4 只使用 `actual` 做最终功率平衡。

## 6. 损耗和辅机责任边界

| 项目 | 唯一责任模块 | 进入 4.4 的方式 |
| --- | --- | --- |
| 风机阵列、光伏场内电缆、潮流能集电损耗 | 4.3 | 已从毛功率中扣除，不再作为 4.4 负荷 |
| 风机、光伏、潮流能设备辅机 | 4.3 计算，4.4 记账 | `pSourceAuxLoad` 独立负荷，仅扣一次 |
| 公共升压、变换、母线及岛级公用系统损耗 | 4.4 | `pPostPOILoss`、`pCommonAux` |
| 电池充放电效率损耗 | 4.5 | 体现在 SOC 状态方程，不在 4.4 再扣 |
| 电解槽 SEC 及制氢 BOP | 4.5 | 统一折算为 `pElectrolyzerActual` |
| 压缩、纯化、储氢循环能耗 | 原则上 4.5 | 若 SEC 已包含，禁止再单列 |
| IT 设备用电 | 4.6 | 形成 `pIT` |
| 冷却、泵、网络、存储及算力辅助用电 | 4.6 | 与 IT 合成 `pDCFacility` |
| 海缆输电损耗 | 4.7 | 送端功率进入 4.4，受端电量用于收益 |
| 管道和船运氢损耗 | 4.7 | 只影响最终交付量，不影响 4.5 产氢量 |
| 海洋用能设备自身电耗 | 4.7 | 作为 `pMarineActual` 进入 4.4 |

必须避免的重复计算：

- 4.7 不能再次根据 PUE 从设施功率重算 IT 功率，应直接读取 4.6 输出。
- 4.7 不能再次根据电解功率计算产氢和储氢状态，应直接读取 4.5 输出。
- `marine.aux_load` 必须明确是海洋服务负荷，不能与 4.3 源侧辅机或 4.4 公共辅机重名。

## 7. 时间尺度和聚合规则

建议冻结四层时间尺度：

| 层级 | 统一步长 | 主要模块 | 用途 |
| --- | --- | --- | --- |
| 原始源侧仿真 | 5 min | 4.3 | 风光潮波动、状态机、预测误差 |
| 主调度层 | 1 h | 4.3--4.9 | 168 h/8760 h 联合运行、经济性和容量评价 |
| 日内精细校核 | 15 min | 4.3--4.7 | 爬坡、短时储能和柔性负荷校核 |
| 构网动态层 | <= 1 s | 4.4/4.5 动态模型 | 频率、电压、虚拟惯量、故障和黑启动 |

联合模型主入口统一读取 1 小时时序。5 分钟、15 分钟和秒级模型只承担校核与参数回灌。

聚合规则：

- 5 分钟功率转 1 小时：按电量守恒求平均功率。
- 电量、产氢量、服务量：求和。
- SOC、氢库存、算力队列：使用时段末状态。
- 可用率：按实际可用时长或状态方程处理。
- 停机、故障、复归：保留事件标签，不线性插值。
- 最大/最小功率、爬坡：分别保存极值，不能只保存平均值。
- 价格、需求：按其原始结算口径聚合。

## 8. 统一基准方案

主基准只能采用一个参考场址：

- 同一海域。
- 同一水深和离岸距离。
- 同一登陆点。
- 同一公共平台拓扑。
- 同一海缆路径长度。

场址未确定前可使用 `SITE_REF_V1` 虚拟参考场址，但必须标注：

```text
[假设场址，仅用于模型联调，不形成工程投资结论]
```

所有比较方案必须使用同一套同步资源数据：

- 同一个代表年。
- 同一风速、辐照、潮流和海况时序。
- 同一时间戳和时区。
- 同一缺测处理方法。
- 同一预测误差场景。

不能为不同方案分别随机生成更有利的资源曲线。

设备可用率建议拆成逐时外生序列：

```text
availabilityState(t)
derateFactor(t)
maintenanceCode(t)
failureCode(t)
```

“单一外送”和“源储算用协同”必须共享同一源侧故障与检修序列。新增电池、电解槽、算力和输出通道分别使用自己的状态序列，但随机种子固定。

## 9. 源侧装机治理

源侧装机按以下关系确定：

1. 4.3 根据场址资源、国产设备和多源互补性提出候选装机组合。
2. 4.9 在容量规划模式下比较候选组合。
3. 项目组选择一个基准组合。
4. 4.1 将其冻结到 `common_case_v1`。
5. 运营调度比较中，所有方案保持相同源侧装机。

不允许 4.3 在每次联合运行时动态改变装机容量。

当前接口联调可暂时冻结：

```text
34 MW 风电 + 4 MW 光伏 + 2 MW 潮流能 = 40 MW
```

来源类型：

```text
[现有公共基线代码假设值，待工程尺度校准]
```

正式工程基准仍应保持“漂浮式风电 85%+、漂浮式光伏互补、潮流/波浪能预留”的项目定位，但具体比例由场址与优化结果决定。

## 10. 海缆既有/新建治理

4.7 负责提供：

- 海缆技术方案。
- 送端容量。
- 受端接纳。
- 距离与损耗模型。
- CAPEX/OPEX 模型。
- 既有/新建两类场景参数。

“使用哪一种作为主基准”由 4.1 冻结，不能由 4.7 自行改变。

主比较采用绿地新建口径：

- 两个方案均视为新建项目。
- 使用相同场址、距离和登陆点。
- 使用同一海缆成本函数。
- 单一外送方案可优化海缆容量。
- 源储算用方案也可优化较小海缆与其他输出通道的组合。

辅助比较采用既有海缆口径：

- 两个方案均使用同一条既有海缆。
- 容量和损耗相同。
- 已发生 CAPEX 作为沉没成本，不计入两方案增量比较。
- 只比较运营收益、弃电、可靠性和新增设备投资。

禁止出现“单一外送按新建海缆、协同方案按既有海缆”的混合比较。

## 11. 与当前 Python 包的映射

当前 `submodules/src/bluehub_submodules` 是可审计子模型原型：

- `power_export.py` 对应 4.7 电力外送链路。注意 `exported_power_mw` 是海缆送端，`delivered_power_mw` 是陆上受端。
- `compute_load.py` 对应 4.6 绿色算力。PUE 换算只在这里做。
- `hydrogen_output.py` 对应 4.5 制氢、储氢和输出请求的可行性。产氢和库存状态只在这里做。
- `marine_load.py` 当前表示海洋服务负荷，不得与源侧辅机或公共辅机混用。
- `balance.py` 当前承担小时级公共母线功率平衡，等价于 4.4 的简化原型。
- `scenario.py` 的贪心调度只用于冒烟测试，不代表 4.9 优化调度。

后续修改代码时，应优先让字段名、单位和测点对齐本文件，而不是沿用含糊的历史变量名。
