## 1. 优化依据

本文件基于当前项目材料与 `往年参考` 目录中的 Blue Hub S8 模型仓库整理，重点参考：

- `往年参考/blue-hub-s8-v0.10.0/configs/technology_parameters.csv`

往年方案的核心优点不是“模型复杂”，而是：

- 物理守恒优先，所有电量每小时只有一个去向。
- 电力、氢气、算力分别记账，避免同一度电重复创造多份收益。
- 海缆物理容量和陆上接纳能力分开，不把“海缆扩容”等同于“消纳能力提升”。
- 算力用 `MWh-IT` 计量，PUE 只负责把 IT 工作量换算成数据中心设施用电。
- 氢气用综合电耗 `SEC` 或等效效率换算，不重复叠加压缩、BOP 等能耗。
- 所有参数有单位、来源等级和不确定性范围。
- 方案比较使用同一资源、同一价格、同一成本边界。
- 先跑固定容量候选方案，再做多情景风险回放，不用事后最优容量冒充真实投资能力。

## 2. 当前方案需要优化的地方

### 2.1 不能只写“多形态输出更优”

当前方案容易写成：

```text
电力外送 + 算力 + 制氢 = 消纳率更高 = 经济性更好
```

这个逻辑不够严谨。往年 S7/S8 的结论说明：高消纳率不一定带来高经济性。容量冗余方案几乎可以消除弃电，但固定资产成本过高时，经济表现可能不如规模适中的联合方案。

优化后应改为：

```text
多形态输出的价值成立条件 =
登陆接纳受限
+ 算力/氢气合同价格足够
+ 工程成本可控
+ 风险情景下表现不劣化
```

### 2.2 不能把海缆容量和陆上消纳能力混在一起

当前公式写成：

```text
0 <= P_grid,t <= min(P_grid_max, P_grid_accept_max)
```

这个方向是对的，但后续要明确区分：

- `C_tx`：海缆物理容量
- `A_tx,t`：海缆可用率
- `G_accept,t`：陆上登陆点逐时接纳能力
- `P_send,t`：海上送端功率
- `P_land,t`：陆上实际接收功率

优化后的约束建议写成：

```text
0 <= P_send,t <= C_tx * A_tx,t
0 <= P_land,t <= G_accept,t
```

如果暂时没有逐时接纳曲线，可以先用：

```text
G_accept,t = P_grid_accept_max
```

但报告里必须说明这是简化假设，后续要用企业或电网侧数据校准。

### 2.3 海缆损耗不要长期停留在线性比例

Day1-Day3 可以先用：

```text
P_grid_del,t = P_grid,t * (1 - loss_cable)
```

但优化版第四章建议加入“可升级的二次损耗代理”，用于体现 I²R 损耗随负载升高而增加：

```text
P_term_loss,t = lambda_term * P_send,t

P_cable_loss,t =
lambda_100 * (D / 100) * (P_send,t^2 / C_tx)

P_land,t =
(1 - lambda_term) * P_send,t
- lambda_100 * (D / 100) * (P_send,t^2 / C_tx)
```

其中：

- `lambda_term`：换流/终端损耗率
- `lambda_100`：每 100 km 满载海缆损耗率
- `D`：离岸距离，km
- `C_tx`：海缆额定容量，MW

如果代码实现来不及，至少在第四章写成“进阶损耗模型，当前最小样例先用线性损耗”。

### 2.4 算力模型应从“功率负荷”升级为“服务交付”

当前方案把算力写成：

```text
P_compute,t = P_it,t * PUE
R_compute,t = P_it,t * price_compute
```

这可以作为基础，但往年方案更严谨：算力服务以 `MWh-IT` 计量，数据中心设施用电由 PUE 换算：

```text
X_IT,t = X_rigid,t + X_flex,t
P_DC,t = PUE_t * X_IT,t / Δt
```

建议当前方案保留两类算力任务：

- 刚性任务：必须当小时完成，未完成计 SLA 惩罚。
- 柔性任务：可延迟到后续小时完成，受最大延迟和队列容量约束。

如果 Day1-Day7 时间不够，可以先做简化版：

```text
0 <= X_compute,t <= C_IT * Δt
P_compute,t = PUE * X_compute,t / Δt
R_compute,t = X_compute,t * price_compute
```

注意：光缆不要进入电力损耗模型。光缆只约束算力服务交付能力：

```text
0 <= X_compute,t <= C_fiber_service * A_fiber,t * Δt
```

### 2.5 氢气模型应使用综合电耗或清晰效率口径

当前方案写成：

```text
H_prod,t = P_h2_el,t * eta_el
```

这可以保留，但要明确 `eta_el` 的单位。如果采用往年模型的方式，更建议写成综合系统 SEC：

```text
H_prod,t = 1000 * P_h2_el,t * Δt / SEC_H2
```

其中：

- `P_h2_el,t` 单位 MW
- `Δt` 单位 h
- `SEC_H2` 单位 kWh/kg

这样可以避免 kg/kWh、%、kWh/kg 混用。

重要口径：

- 如果 `SEC_H2` 已包含电解槽和 BOP 综合电耗，就不要再重复叠加常规辅助功耗。
- 压缩、液化、转氨、船运等能耗要作为单独运输链路参数说明。
- 未销售库存不能计收入。
- 储氢期末状态建议闭合，即 `H_T = H_0`，防止模型靠年末囤氢虚增价值。

### 2.6 经济性必须分三层

往年方案将价值分成三层，当前方案建议直接采用：

| 层级 | 含义 | 用途 |
|---|---|---|
| 运行毛利 | 电力、氢气、算力收入减去变动成本 | 判断调度逻辑是否合理 |
| 灵活资产增量净值 | 运行毛利减去算力、电解槽、储氢、管道等灵活资产年化成本 | 判断增建灵活设施是否划算 |
| 完整项目年化净值 | 再扣除风电、海缆、换流、平台、登陆侧工程等公共成本 | 判断整体项目是否财务可行 |

报告里必须避免把“相对直接送电多赚”写成“完整项目盈利”。

## 3. 优化后的总体建模路线

当前 Day1-Day7 建议改成以下路线：

```text
S0：电力外送基线
    风/光出力 -> 关键负荷/海洋用能 -> 海缆外送 -> 弃电

S1：加入制氢与储氢
    富余电力 -> 电解制氢 -> 储氢 -> 管道/船运销售

S2：加入绿色算力
    富余电力 -> 数据中心设施用电 -> MWh-IT 服务 -> 光缆交付

S3：集成调度
    电力外送、制氢、算力、海洋用能共享同一个海上功率平衡

S4：候选方案回放
    直接送电、算力主导、制氢主导、联合合同、容量冗余、保供储备

S5：风险与敏感性
    登陆受限、低风、算力价格偏弱、氢价偏弱、上行情景、海缆/光缆故障
```

Day1-Day7 不一定全部实现 S5，但方案设计要按这个结构搭，避免后期推倒重来。

## 4. 优化后的候选方案

不要再只写“单一外送、均衡分配、算力优先、制氢优先”。建议改成更可审计的候选方案集：

| 方案编号 | 方案名称 | 核心含义 | 验证的问题 |
|---|---|---|---|
| C0 | 直接送电基准 | 不建设算力和制氢，仅通过海缆外送 | 单一外送在当前登陆约束下表现如何 |
| C1 | 算力主导 | 建设一定规模 IT 容量和光缆服务能力 | 可移动算力能否吸收短时富余电力 |
| C2 | 制氢主导 | 建设电解槽、储氢和管道/船运能力 | 氢气能否承担长时价值转化 |
| C3 | 联合合同 | 中等规模算力 + 中等规模制氢 | 算力和氢气能否分担价格和消纳风险 |
| C4 | 容量冗余 | 取算力主导和制氢主导容量并集 | 追求极高消纳率是否会因成本过高而不划算 |
| C5 | 保供储备 | 储氢 + 燃料电池，仅用于低风/故障保供 | 商品制氢和可靠性保供是否应分账 |

## 8. 第五章写法优化

第五章不要只比较“哪个收益最高”，而应回答 4 个问题：

1. 在登陆接纳受限时，多形态输出能减少多少弃电？
2. 在算力和氢气合同价格达到什么水平时，灵活设施相对直接送电才划算？
3. 在低风、弱合同、登陆受限等情景下，哪个方案风险更小？
4. 商品制氢和保供储氢是不是同一个价值逻辑？

建议指标：

| 指标 | 含义 |
|---|---|
| 新能源利用率 | 消纳改善 |
| 弃电量 | 资源浪费程度 |
| 电力交付量 | 外送能力 |
| 算力服务量 | MWh-IT |
| 制氢量/售氢量 | kg |
| 运行毛利 | 不含固定资产 |
| 灵活资产增量净值 | 含灵活资产年化成本 |
| 完整项目年化净值 | 含公共工程成本 |
| 最差情景收益 | 下行风险 |
| 后悔值 | 与情景最优方案的差距 |
| EENS/缺供电量 | 可靠性 |

## 9. 企业调研问题优化

企业交流前，你负责的问题应更具体：

### 电力与登陆接纳

- 拟定场址到登陆点的海缆路由距离是多少？
- 海缆电压等级、交流/直流方案和额定容量如何设定？
- 登陆点是否有逐时或典型日接纳能力曲线？
- 当前瓶颈在海缆、换流站、登陆站，还是陆上电网消纳？
- 海缆故障或检修时，是否允许岛上负荷继续运行？

### 算力与光缆

- 算力业务是 AI 训练、离线渲染、数据备份，还是实时业务？
- 可迁移任务比例是多少？
- 最大允许延迟是多少？
- 客户愿意按什么口径付费：元/GPU小时、元/kWh-IT、元/任务？
- 光缆容量、可用率和 SLA 如何定义？
- 数据中心 PUE 是否有海上/海底工程依据？

### 氢气与储运

- 输出产品是气态氢、液氢、氨、甲醇，还是多方案对比？
- 外部需求是逐时稳定提货，还是按日/周/月提货？
- 到岸价格、纯度、压力、交付点在哪里？
- 管道输氢是否有路径和管径设想？
- 船运是否需要液化、转氨或压缩？
- 储氢是否承担保供功能，还是只为商品销售服务？

### 成本与收益

- 海缆、换流、登陆工程由谁投资？
- 算力设施、光缆、平台空间是否计入本项目 CAPEX？
- 氢气储运设施由能源岛侧还是用户侧投资？
- 是否存在容量费、最低采购量、长期合约或保底价格？

## 10. 最重要的表述调整

当前报告建议避免以下表述：

```text
多形态输出一定提高经济性。
制氢可以无限消纳弃电。
算力通过光缆传输电能。
海缆扩容就能解决消纳问题。
消纳率越高，方案越优。
储氢销售价值和保供价值可以直接相加。
```

建议改成：

```text
多形态输出在登陆受限、合同价格明确、工程成本可控时，可以提升新能源利用率并改善相对直接送电的经济性。
制氢受电解槽容量、储氢容量、输氢能力、外部需求和储运成本共同约束。
算力价值来自可迁移数字任务在岛上消纳电力，并通过光缆交付服务结果。
海缆扩容只有在登陆侧接纳能力同步提高时才可能改善消纳。
高消纳率需要与固定资产成本一起评价。
商品制氢和保供储氢应分账核算。
```

## 11. 一句话优化结论

当前方案应从“源储算用概念方案”升级为：

```text
一个以登陆约束为核心矛盾、以电力/算力/氢气分账为基础、以固定候选方案和多情景回放验证稳健性的深远海能源岛方案。
```

这样写，比单纯强调“多能互补、协同调度、价值最大化”更可信，也更接近往年参考方案的成熟度。
